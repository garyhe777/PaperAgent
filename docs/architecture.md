# PaperAgent 架构说明

## 1. 整体数据流

PaperAgent 的主流程分成四段：

1. CLI 或 Web 入口接收用户请求。
2. 服务层负责业务逻辑，比如导入论文、检索、问答、生成 PPT。
3. 存储层负责把论文元数据和分块内容写入 SQLite，把向量索引写入 Chroma，把原始文件保存在文件系统。
4. 表示层把结果渲染成命令行输出、API 响应或前端页面。

这样设计的原因很简单：入口层只负责“接请求”，服务层只负责“做事情”，存储层只负责“保存数据”。初学者可以把它理解为把一大段难懂代码拆成几层小积木。

## 2. 为什么要分层

- `cli`：命令行入口。它只解析命令行参数，然后调用服务层。
- `storage`：数据库和仓库。它负责保存 `paper_id`、标题、PDF 路径、Markdown 路径、chunk 内容等。
- `ingest`：导入流程。它把 PDF 或 URL 变成 Markdown 和 chunk。
- `retrieval`：检索流程。它同时做向量检索和 BM25 检索，再把结果融合。
- `agent`：问答 Agent。它决定什么时候检索，然后基于证据回答。
- `ppt`：PPT 生成。它先整理大纲，再渲染为 `.pptx`。

如果以后要加前端，我们可以直接复用服务层，而不用重写导入、检索和 Agent 逻辑。

## 3. 配置对象的作用

`Settings` 负责管理路径、模型地址、API key、数据库位置等配置。这样做有两个好处：

1. 不需要把路径和密钥写死在代码里，部署时更安全。
2. CLI、测试、FastAPI 都可以读取同一套配置，避免每个地方各写一份。

初学者可以把它理解成“程序启动时统一读一张配置表”，后面的模块都从这里拿设置。

## 4. `paper_id` 是怎么串起全链路的

每篇论文都会分配一个 `paper_id`。它会同时出现在：

- SQLite 的 `papers` 表
- SQLite 的 `chunks` 表
- 文件系统中的 PDF 和 Markdown 文件名
- Chroma 元数据
- BM25 索引文件
- PPT 输出文件名

这就是为什么系统能知道“这条 chunk 属于哪篇论文”“这次聊天应该检索哪篇论文”“这个 PPT 是为哪篇论文生成的”。

## 5. 事件流为什么重要

CLI 聊天时，我们不仅要看到最终答案，还想看到“正在检索”“命中了哪些片段”“正在生成回答”。所以 Agent 输出分成两条线：

- 一条是最终答案的流式文本
- 一条是过程事件，比如 `tool_called`、`rag_hit`

以后做 Web 时，前端只要监听同样的事件流，就能复用这套机制。
